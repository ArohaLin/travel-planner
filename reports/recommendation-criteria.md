# 推薦精選・建置準則

> 適用範圍：所有區域的精選推薦建置（台東已套用；嘉義等後續一律遵循）。
> 目標：以**真實資料**為底、**人工策展**把關，產出「可比較、可信、誠實」的精選清單；低 token、可按需重建。

## 一、資料模型（靜態策展 + 即時補事實）

- **靜態存**（一次建好、幾乎不變）：`google_place_id`、名稱、分類、座標、**推薦理由**、標籤、來源徽章、可信度分數。
- **顯示時即時補**（易變、不入庫）：評分、評論數、營業時間、照片 → 用 `place_id` 當下向 Google 取。
- 好處：選哪些、為什麼是靜態的；會過時的數字即時抓 → **不過時、零維護，也符合 Google 條款**（長期只存 place_id）。

## 二、建置流程（三階段；前兩段不耗 AI）

| 階段 | 做什麼 | 耗 AI？ |
|---|---|:--:|
| A 蒐集 | 多組關鍵字向 Google Places 查真實地點，只收該區；同點被多查詢命中記 hits。再對「有名卻沒搜到」的點定向補查 | 否 |
| B 評分 | 用信賴下界／貝氏校正算分（見下），程式初篩入圍 | 否 |
| C 策展 | 人工逐筆把關：取捨準則（見第四節）、寫誠實短評/標籤/徽章 | 對話內 |

> 低 token 原則：蒐集/查證/算分/初篩全用程式；AI 只在最後策展，且批次處理入圍者。**瀏覽推薦時 0 AI token**（只即時打 Places）。

## 三、量化評分（信賴下界 / 貝氏校正）

不直接用星等，避免「5.0（8 則）」少量灌水沖上來：

```
貝氏 = (v/(v+M))·R + (M/(v+M))·C
R=星等  v=評論數  C=基準分 4.2  M=平滑常數 120
```

評論越少 → 分數越往 4.2 拉；評論越多星等才越算數。貝氏只負責**排序**，不負責「收不收」。

## 四、取捨準則：代表性 vs 灌水（核心）

**一句話：灌水 ≠ 淘汰；灌水 = 不採信「星等」。**
把兩件事拆開——「灌水」是**資料可信度**問題（只該讓我們不信那個星等），「代表性／真品質」才是**值不值得推薦**問題。

**先問「值不值得推薦（要有獨立證據）」，再問「星等可不可信」：**

| | 星等可信 | 星等灌水 |
|---|---|---|
| **有代表性／真品質** | ✅ 收，正常呈現 | ✅ 收，但**不靠星等**——以名產/地標/跨來源理由收錄，**誠實標註星等灌水** |
| **普通／無代表性** | 看名額，後段可不收 | ❌ **不收**——星等不可信又無其他入選理由 |

**關鍵一問（決定能不能用代表性救它）**：
> 除了被灌水的星等以外，還有沒有「別的、可信的」入選理由？

- 有（公認名產、官方收錄、多家媒體、必比登、多位獨立部落客…）→ **收**，並誠實標註星等灌水。例：地瓜酥、麻糬。
- 沒有（賣點幾乎只剩可疑的高星等）→ **不收**。例：高分燒肉、輕食。

**紀律**：代表性必須有**獨立證據**，不能只憑「我覺得它很紅」，以免「有名就放行」被濫用。

## 五、防灌水偵測訊號

- 星等 ≥4.9 且評論量大（典型「好評送贈品」指紋）。
- 罐頭式好評扎堆、評論出現「打卡五星送…」字眼。
- 評論數異常爆衝、評價兩極（大量 5★ 配一群 1★）。
- 少量評論的高星（用貝氏下界壓低）。

## 六、呈現原則

- 卡片**不主打星等**：以「為什麼推薦＋來源徽章＋評分(評論數)」呈現，星等只是配角且一定附評論數。
- 灌水但入選者：用名產/地標徽章＋誠實短評，星等註明「含好評活動成分」，**避免使用者被灌水數字誤導**。

## 七、維護模式

平常零維護。需要時（資料舊了、要去新地方）再「按需重建」某區域；沒有背景排程、爬蟲或後台。
